Invigilator

1. Introduction

1.1 Purpose

The purpose of this document is to present a detailed description of the open-source software Invigilator. It will explain the purpose and features of the software, the interfaces of the software, what the software will do and the constraints under which it must operate. This document is intended for users of the software and also potential developers.

1.2 Document Conventions

This Document was created based on the IEEE template for System Requirement Specification Documents.

1.3 Intended Audience and Reading Suggestions

Educators to take online tests and students who will be attending the online tests.

1.4 Production Scope

Invigilator is web application for taking online tests. It can be used for supervising and marking the exams more efficiently. There will be ways of answering questions in many ways. It will also benefit the review process of the tests.

2. Overall Description

2.1 Product Perspective

The Invigilator web application aims to simplify online tests, benefiting both examiners and examinees, while also ensuring the integrity of the examination process. It saves time, enhances efficiency, and provides a user-friendly experience, contributing to the effectiveness of online education.

2.2 Product Functions

  • There should be a profile for examiners
  • There should be a profile for examinees
  • Multiple types of answering systems
  • Automatic evaluation for closed type answers
  • Exams availability can be limited by the examiner
  • Exam can have a set duration
  • There will be review system for an evaluated exam
  • Their will certain restrictions to examinees device for the duration of the exam
  • The webcam should be used to keep an eye on the examinee
  • Seamless switch between devices

2.3 User Classes and Characteristics

  • Educators/Examiners who will take the test
  • Students/Examinee who will be giving exams

2.4 Operating Environment

The software will be operated on the following platforms:

  • Windows 7 or higher
  • Android 7.0 (Nougat) or higher

The software will require the following hardware:

  • Working webcam
  • Working microphone
  • Working internet connecton

The software will also require the following software:

  • A web browser
  • A supported operating system

The browser compatibility, screen resolution, keyboard and mouse, and accessibility considerations are still applicable.

2.5 Design and Implementation Constraints

  • The platform will be designed to work seamlessly on modern web browsers, including Google Chrome and Microsoft Edge
  • The user interface will be responsive, ensuring usability across various devices, from desktops to smartphones
  • Security measures, such as data encryption and secure authentication, will safeguard exam content and user data security measures, such as data encryption and secure authentication, will safeguard exam content and user data
  • Application may need system administration permission to implement method related with exam integrity
  • Must be able to handle usersâ€™ profile on multiple devices seamlessly

2.6 User Documentation

  • In app instruction for users
  • A FAQ section
  • A helpline email

2.7 Assumptions and Dependencies

Some computer protection software might interfere with the platform's functionality. Ensuring proper access permissions and compatibility with such software is necessary for uninterrupted use.

3.2 Hardware Interfaces

3.2.1 User Devices

The platform interacts with the following user devices:


Desktop and Laptop Computers: Users access the platform through web browsers on desktop and laptop computers. The platform should be compatible with a variety of screen sizes and resolutions.


Smartphones and Tablets: The platform is designed to be responsive and accessible on smartphones and tablets. Users can take exams and review results on these devices.

3.2.2 Cameras (Webcams)

The platform may require access to integrated or external cameras (webcams) on user devices for exam monitoring purposes. Cameras enable remote proctoring to maintain exam integrity

3.2.3 Network Connectivity

The platform relies on network connectivity for the following purposes:


Internet Connection: Users require a stable Internet connection to access the platform, take exams, and submit answers.


Server Communication: The platform communicates with the server infrastructure to retrieve exam questions, submit answers, and receive results.


3.2.4 Printing Devices

While not a primary interface, users may choose to print exam questions or results. The platform should allow users to generate printer-friendly versions of exam content.

3.2.5 Input Devices

The platform may require access to integrated or external cameras (webcams) on user devices for exam monitoring purposes. Cameras enable remote proctoring to maintain exam integrity

3.3 Software Interfaces

3.3.1 Web Browsers

The platform is accessed by users through modern web browsers, including but not limited to:

  • Google Chrome
  • Microsoft Edge

    • The platform's user interface and functionality should be optimized for these browsers to ensure a consistent experience across different platforms

    3.3.2 Server Infrastructure

    The platform interacts with the server infrastructure to manage various operations, including:

  • Serving exam content to users
  • Collecting and storing user answers
  • Calculating exam results
  • Managing user accounts and authentication

    • The platform relies on secure communication protocols (e.g., HTTPS) to exchange data between users' devices and the server.

    3.3.3 Third-Party Services

    The platform may integrate with third-party services for specific functionalities, such as Authentication services for user login. Integration with third-party services requires adherence to their respective APIs and guidelines to ensure compatibility and smooth operation.

    3.3.4 Operating Systems

    The platform is designed to operate on various operating systems, including but not limited to:

  • Windows 7 or higher
  • Android 7.0 or higher

    • The platform should be compatible with the user devices' operating systems, allowing users to access the platform seamlessly.

    3.4 Communications Interfaces

    3.4.1 HTTP/HTTPS

    The platform communicates with users' web browsers and server infrastructure using the Hypertext Transfer Protocol (HTTP) and the secure variant, Hypertext Transfer Protocol Secure (HTTPS). HTTP/HTTPS is utilized for:

  • Delivering exam content and questions to users' devices
  • Receiving users' submitted answers
  • Displaying results and exam information
  • The use of HTTPS ensures secure data transmission and protection of sensitive user information.

    • 3.4.2 APIs (Application Programming Interfaces)

    The platform may interact with third-party services through APIs. APIs provide a standardized way to exchange data and perform actions with external services. The platform adheres to the APIs provided by third-party services for user authentication and login through external identity providers.

    3.4.3 Email Communication

    The platform communicates with users through email to provide:

  • Account-related notifications
  • Exam reminders
  • Support-related correspondence
  • Email communication ensures users are informed about important updates and activities related to their use of the platform

    • 4. System Features

    4.1 Authentication

    The platform ensures secure access using authentication mechanisms like username-password combination or multi-factor authentication

    4.2 Timing

    Exam has a duration as well as a validation period.

    4.3 Cohorts

    Cohorts are groups of students that can be created by the educator. The educator can assign exams to cohorts.

    4.4 Question type

    There are multiple types of questions that can be used in an exam.Such as multiple choices, text based, and also voice based.

    4.5 Marking

    Any closed-type answer can be automatically marked.

    4.6 Dispute System

    Examinees can call disputes about their marks. It notifies the examiner asking for a resolve.

    4.7 Supervisation

    Examinee should be supervised via webcam. Any other person on the frame will notify the examiner. If the examinee is not present on the frame it will also notify the examiner

    4.8 Restriction

    The examinee's device will have certain restrictions during the exam. The examinee should not be able to access any other tabs during the exam. If not possible then the examiner should be notified of such occurance.

    4.9 Seamless Switching

    The examinee should be able to switch devices during the exam. The examinee should be able to continue the exam from where he/she left off.

    5. Other Nonfunctional Requirements

    5.1 Performance Requirements

    5.1.1 Response Time

    Page Loading: The platform should load pages and exam content within 2 seconds on average, with a maximum loading time of 5 seconds.


    Question Navigation: Moving between questions within an exam should take no more than 1 second.

    5.1.2 Database Access

    Database queries should be optimized to ensure that data retrieval and storage operations are completed within 500 milliseconds.

    5.1.3 Network Latency

    The platform should account for network latency and aim to minimize the impact of latency on user interactions. Network requests should be designed to handle latency of up to 300 milliseconds.

    5.1.4 Device switching

    The web application should handle network failure and let the user switch to a different device seamlessly without losing any ongoing progress.

    5.2 Safety Requirements

    User data should be protected. Any unauthorized access should be prevented. All-access needs to be recorded. Any measure that needs to be implemented to ensure exam integrity will be implemented. Data of profile and ongoing works needs to be backed up properly to ensure there are no data loss.

    5.3 Security Requirements

    Some supervising features may require administrator access from the examinee device for the purpose of supervising.

    5.4 Software Quality Attributes

    Invigilator will be easy to use and will only require basic computer skills.

    Appendix A: Glossary

    API: Abbreviation for Application Programming Interface. A set of rules and protocols that allows different software applications to communicate and interact with each other.


    Authentication: The process of verifying the identity of a user, typically through the use of a username and password or other authentication methods.


    Concurrency:The ability of a system to handle multiple tasks or processes simultaneously.


    Encryption:The process of converting data into a code to prevent unauthorized access.


    HTTP:Abbreviation for Hypertext Transfer Protocol. A protocol used for transferring data over the internet.


    HTTPS:Abbreviation for Hypertext Transfer Protocol Secure. A secure version of HTTP that uses encryption to protect data.


    Multi-factor Authentication (MFA): A security process that requires users to provide multiple forms of identification before accessing a system.


    Responsive Design:Design approach that ensures a consistent user experience across different devices and screen sizes.


    Role-Based Access Control:A method of restricting access to system resources based on the roles of individual users.


    SSL/TLS:Abbreviations for Secure Sockets Layer and Transport Layer Security. Protocols used to establish secure and encrypted connections between a user's device and a server.


    Web Browser:Software used to access and view websites on the internet.
